# AGENTS.md

## Project
Babushkiberpunk Generator

Статический генератор для НРИ «Бабушкиберпанк» на React + Vite + TypeScript.
Проект рассчитан на публикацию через GitHub Pages и должен работать без бэкенда, базы данных и внешних API.

## Main goals
Сделать простой, понятный и поддерживаемый генератор с двумя основными экранами:

1. Генератор бабушки (лист персонажа)
2. Генератор приключения (зацепка + 3 сцены)

Также есть дополнительная таблица частушек, не являющаяся частью основных генераторов.

## Hard constraints
- Это полностью статический сайт.
- Никаких серверов, БД, аутентификации, облачных функций и API.
- Не добавлять лишние зависимости без явной необходимости.
- Не использовать UI-фреймворки без прямой необходимости.
- Не использовать state management библиотеки вроде Redux, MobX, Zustand, если задача решается штатными средствами React.
- Не использовать роутинг, если задача решается вкладками внутри одного приложения.
- Не добавлять drag-and-drop, анимации, сложные визуальные эффекты и декоративные библиотеки без явной причины.
- Не изобретать собственную игровую механику. Генерация должна строго следовать ролл-таблицам из книги.
- Не заменять контент таблиц выдуманным содержимым.
- Не хардкодить значения таблиц в компонентах, если они уже лежат в JSON.
- Не смешивать данные, логику генерации и React UI в одном месте.

## Architecture principles
Проект должен быть data-driven.

### Separate concerns
Нужно строго разделять:
- данные таблиц;
- логику бросков и генерации;
- UI-компоненты;
- стили.

### Data source
Контент таблиц хранится в JSON-файлах в `src/data/...`.

Код должен читать эти файлы как источник истины.
Если структура данных уже описана, не дублировать её в коде без причины.

### Re-roll support
Интерфейс должен поддерживать:
- полную генерацию сущности целиком;
- перегенерацию отдельных элементов.

Примеры:
- в генераторе бабушки можно отдельно перегенерировать имя, отчество, прошлое, вещчь, коронный номер и т.д.;
- в генераторе приключения можно отдельно перегенерировать зацепку, сцену целиком или её часть.

### Deterministic structure
Структура экранов известна заранее и не должна строиться как абстрактный no-code конструктор.
Нужен аккуратный прикладной код под конкретный продукт, но с использованием данных из JSON.

## UI structure
### Top-level screens
В приложении есть две основные вкладки:
- Бабушка
- Приключение

Дополнительно может быть отдельный небольшой блок или вкладка для частушек, но это вторично.

### Layout pattern
Для обоих основных экранов:
- в верхней части страницы отображается результат генерации;
- в нижней части страницы отображаются табы с ролл-таблицами, относящимися к текущему экрану.

### Character screen
Экран генератора бабушки показывает лист персонажа.

Важные особенности:
- имя и отчество роллятся отдельно, но отображаются как единое поле;
- цвет волос может использоваться как скрытый служебный параметр для будущего портрета;
- характеристики не роллятся по таблице, а распределяются вручную по правилам;
- если в базовом листе нет поля, но оно нужно для генератора, допустимо аккуратно добавить его в вёрстку.

### Adventure screen
Экран генератора приключения показывает:
- одну зацепку;
- три сцены.

Каждая сцена состоит из:
- места действия;
- технологии / киберпанка;
- врагов народа.

Враги народа имеют составную структуру:
- внешний вид / маскировка;
- истинный тип;
- подтип, выбираемый по связанной подтаблице.

## Data format conventions
Следовать уже принятому формату JSON.

### Simple keyed tables
Для обычных d6 или d66 таблиц:
- `key` хранится как строка, например `"1"` или `"11"`.

### Range tables
Для диапазонов:
- `range` хранится как массив чисел, например `[11, 13]`.

### Entries
Если запись таблицы содержит краткое название и описание, использовать:
- `label`
- `description`

Если запись представляет собой единый цельный текст, допустимо использовать только `label`.

### Highlights
Если в тексте есть важные термины, которые должны выделяться в интерфейсе, использовать `highlights`, а не HTML внутри строки.

Не вставлять HTML-теги вроде `<b>` или `<i>` в JSON-тексты без крайней необходимости.

## Highlight rendering
Некоторые слова и фразы должны стилизоваться в UI через структурированные данные.

Примеры категорий:
- характеристика;
- преимущество;
- помеха;
- критический успех;
- критическая неудача;
- ход ведущего против игроков.

Код должен рендерить эти выделения безопасно и предсказуемо.
Не использовать `dangerouslySetInnerHTML` для таких задач, если можно обойтись обычным React-рендерингом.

## Code style expectations
- Использовать TypeScript.
- Предпочитать понятный и прямой код вместо “умной” переархитектуренной магии.
- Писать небольшие чистые функции для логики бросков и поиска результатов в таблицах.
- Избегать чрезмерной абстракции.
- Избегать premature optimization.
- Давать переменным и функциям понятные имена.
- Предпочитать композицию небольших компонентов.
- Не делать огромные универсальные компоненты “на все случаи жизни”.

## Recommended implementation approach
При добавлении фич:
1. Сначала определить, есть ли нужные данные в JSON.
2. Затем написать или переиспользовать чистую функцию генерации.
3. Затем подключить это к UI.
4. Затем добавить точечный стиль и обработку reroll.

## Roll logic expectations
Нужны утилиты для:
- d6;
- d66;
- поиска результата по `key`;
- поиска результата по `range`;
- обработки таблиц с подтаблицами.

Логика бросков должна быть изолирована от UI и пригодна для повторного использования.

## Styling expectations
- Стили должны быть простыми, чистыми и читаемыми.
- Без визуального шума.
- Без “игрового” перегруза интерфейса.
- Приоритет: удобочитаемость, ясная иерархия, удобные кнопки reroll.
- Допустимо выделять разные характеристики разными цветами.
- Допустимо выделять преимущества, помехи, криты и ходы ведущего цветом и/или курсивом, но без кислотной перегрузки.

## What to avoid
- Не переписывать весь проект без необходимости.
- Не менять структуру данных без веской причины.
- Не вводить сложную конфигурационную систему, если задача решается прямым кодом.
- Не превращать проект в общий генератор для всех НРИ.
- Не строить CMS, редактор контента или административную панель.
- Не создавать лишние уровни абстракции “на вырост”, если они не дают практической пользы прямо сейчас.

## Preferred folder responsibilities
Придерживаться разделения по смыслу, например:
- `src/data` — JSON-таблицы;
- `src/domain` или `src/lib` — логика генерации и бросков;
- `src/components` — переиспользуемые UI-компоненты;
- `src/features` — код конкретных экранов и сценариев;
- `src/types` — типы данных.

Можно предлагать уточнения структуры, но без избыточного усложнения.

## When changing existing code
Если меняется существующий код:
- сохранять текущий стиль проекта;
- не делать несвязанных рефакторингов;
- не трогать части, не относящиеся к задаче;
- по возможности предпочитать минимальные, локальные изменения.

## Definition of done
Решение считается хорошим, если:
- оно работает в статическом React/Vite приложении;
- использует данные из JSON, а не хардкод;
- поддерживает понятный reroll;
- не усложняет архитектуру без пользы;
- остаётся читаемым для человека, который будет дальше руками поддерживать этот проект.
